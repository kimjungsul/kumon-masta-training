# 스프린트 계획 스킬

## 사용 방법
"스프린트 계획 짜줘" 또는 "/sprint-plan [스프린트 번호/기간]" 으로 호출

---

## 실행 순서

### 1단계 — 컨텍스트 수집 (먼저 질문)
- 스프린트 기간은? (기본: 2주)
- 팀 구성 및 인원수는?
- 이번 스프린트 목표는?
- 백로그에서 우선순위가 정해진 항목이 있는가?

### 2단계 — 6단계 스프린트 계획

#### Step 1. 팀 용량 산정
```
총 가용 인원: [N]명
스프린트 기간: [N]일
예외 사항: PTO, 미팅, 장애 대응 등
버퍼: 15~20% 차감
실제 가용 포인트: [N]
```

#### Step 2. 스토리 선정
- 백로그에서 우선순위 높은 순으로 선택
- Definition of Ready 확인:
  - [ ] 요구사항 명확한가?
  - [ ] 공수 산정됐는가?
  - [ ] 의존성 파악됐는가?

#### Step 3. 의존성 매핑
| 스토리 | 의존하는 항목 | 담당 팀 |
|--------|------------|---------|
| | | |

#### Step 4. 리스크 식별
| 리스크 | 영향도 | 대응 방안 |
|--------|--------|----------|
| | 높음/중간/낮음 | |

#### Step 5. 스프린트 계획서
```
## 스프린트 [N] 계획
기간: MM/DD ~ MM/DD
스프린트 목표: [한 문장]

### 커밋된 스토리
| 티켓 | 내용 | 담당자 | 포인트 |
|------|------|--------|--------|
| | | | |

### 총 포인트: [N]
### 팀 용량: [N]
```

#### Step 6. 스프린트 목표 정의
> "이번 스프린트에서 전달할 핵심 가치를 한 문장으로"

### 3단계 — 품질 기준
- 목표는 측정 가능하고 명확하게
- 과부하 금지 — 용량의 80%만 커밋
- Jira 티켓 번호 연동 가능하도록 작성
- 완료 기준(Definition of Done) 명시
